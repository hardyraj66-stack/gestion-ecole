🏠 Module Salles

Salles de l'établissement, types et vérification de disponibilité

Schéma Salle

{
  nom:         String  (requis)
  capacite:    Number  (requis, défaut 30)
  description: String  (défaut "")
  type:        Enum    ['standard','laboratoire','informatique','sport','arts']
}

Endpoints API

MéthodeRouteDescription
GET/sallesToutes les salles
GET/salles/disponibles?jour=Lundi&heure_debut=08:00&heure_fin=09:00Disponibilité
GET/salles/:idUne salle
POST/sallesCréer
PATCH/salles/:idModifier
DELETE/salles/:idSupprimer

Endpoint Disponibilité — Détail

Retourne toutes les salles avec un champ disponible: boolean et, si occupée, les détails de l'occupant :

[{
  id: "...", nom: "Salle 101", capacite: 30, type: "standard",
  disponible: false,
  occupant: {
    classe_id: "...",
    matiere_nom: "Mathématiques",
    heure_debut: "08:00",
    heure_fin: "09:00"
  }
}]

Algorithme de conflit

Deux créneaux sont en conflit si : creneau.salle === salle.nom && creneau.jour === jour && creneau_debut < heure_fin && creneau_fin > heure_debut

Règle métier Socket : pour les salles, les opérations create/update/delete ne mettent pas à jour le state local côté frontend. On laisse le socket arriver pour éviter les doublons.
