🎯 Lazy Loading Frontend

1 page = 1 appel API ciblé, pagination et recherche côté backend

Principe

L'application ne précharge plus globalement les données. Chaque page appelle uniquement l'endpoint read dont elle a besoin.

PageHookEndpoint
DashboardusePageFetch(() => readApi.dashboard(...))/read/dashboard?classesPage=...
ClassesusePageFetch(() => readApi.classesList(...))/read/classes?page=...&search=...
Classe/ÉlèvesusePageFetch(() => readApi.classeEleves(...))/read/classes/:id/eleves?page=...&search=...
ÉlèvesusePageFetch(() => readApi.elevesList(...))/read/eleves?page=...&search=...&classeId=...
MatièresusePageFetch(() => readApi.matieresList(...))/read/matieres?page=...
SallesusePageFetch(() => readApi.sallesList(...))/read/salles?page=...
NotesusePageFetch(() => readApi.notesPage())/read/notes
PlanningusePageFetch(() => readApi.planning())/read/planning
BulletinusePageFetch(() => readApi.bulletin(id,t))/read/bulletin/:id?trimestre=t

Anti double appel

React StrictMode monte les composants deux fois en développement. Le hook usePageFetch utilise isFetchingRef pour empêcher un second appel concurrent.

if (isFetchingRef.current) return;
isFetchingRef.current = true;
...
isFetchingRef.current = false;

Synchronisation temps réel

Après une mutation :

  1. le backend émet un événement Socket.IO
  2. le contexte correspondant appelle notifyDataChange()
  3. les hooks usePageFetch écoutent onDataChange()
  4. ils re-fetchent l'endpoint read après 500ms (temps laissé au ViewBuilder)

Pagination / recherche serveur

Le frontend n'exécute plus paginate(), filter() ou useMemo() pour trier les listes principales. Il envoie uniquement :

  • page
  • limit
  • search
  • classeId (si applicable)

Le frontend reste responsable des états d'interface (page courante, valeur de recherche, classe sélectionnée), mais le backend applique le filtre et la pagination effectifs.
