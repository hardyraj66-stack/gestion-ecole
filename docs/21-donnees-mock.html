Stratégie de fallback

Chaque contexte React suit le même pattern :

const getAll = async () => {
  try {
    const res = await fetch(API_URL + '/xxx');
    if (res.ok) {
      const data = await res.json();
      // Si l'API retourne des données → les utiliser
      if (data && data.length > 0) setItems(data);
      // Sinon → fallback mock
      else setItems(mockXxx);
    } else {
      setItems(mockXxx);  // Erreur HTTP → mock
    }
  } catch {
    setItems(mockXxx);    // Réseau inaccessible → mock
  }
};

L'application est 100% fonctionnelle sans backend. Les données mock permettent de démontrer toutes les fonctionnalités.

Génération des données

Élèves

  • 30 prénoms masculins + 30 prénoms féminins + 40 noms de famille
  • Genre aléatoire 50/50
  • Date de naissance cohérente avec le niveau : 6ème → 2013, Terminale → 2007
  • Email auto-généré : prenom.nom@ecole.fr

Notes

  • Base = 8 + random(0-10), variation ±2 → distribution réaliste centrée sur 13
  • Arrondi au demi-point (step 0.5)
  • 30% de chance d'avoir un commentaire aléatoire

Créneaux

  • Nombre de créneaux par matière proportionnel au coefficient
  • Anti-collision : un set de créneaux occupés par classe
  • Salles spécialisées : PC → Labo Physique, EPS → Gymnase, etc.

Seeder MongoDB

SeederService dans server/src/data/ :

  • Implémente OnModuleInit
  • Vérifie salleModel.countDocuments()
  • Si 0 → insère toutes les données avec les vrais _id MongoDB comme références
  • Si > 0 → skip (« Base déjà peuplée »)

Le seeder et les mocks frontend utilisent les mêmes données source. La seule différence est que le seeder utilise les ObjectId MongoDB pour les références, tandis que les mocks utilisent des IDs string simples.
