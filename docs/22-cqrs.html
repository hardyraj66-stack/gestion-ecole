🔀 Architecture CQRS

Séparation lecture / écriture, collections read dédiées, ViewBuilder, pagination serveur

Principe

L'application applique le pattern CQRS (Command Query Responsibility Segregation) :

  • Les écritures (POST, PATCH, DELETE) passent par les write controllers et écrivent dans les collections write (source de vérité).
  • Les lectures (GET) passent par un unique ReadController et lisent des collections read dénormalisées, pré-calculées, paginées côté serveur.
  • Les deux côtés ne partagent jamais les mêmes collections MongoDB.

Flux complet

ElevesService.create() → INSERT dans "eleves" (write) │ ├── 2. EventsGateway.emit() → Socket.IO: eleve:created │ └── 3. ViewBuilder.onEleveWrite() → PROJECTION dans "read_eleves" + "read_classes" │ │ └── Socket.IO: eleve:created │ ▼ EleveContext → notifyDataChange() │ ▼ usePageFetch → setTimeout(500ms) → re-GET /read/eleves → page mise à jour

Write Controllers

Les write controllers n'exposent que POST, PATCH, DELETE (pas de GET). Après chaque mutation :

  1. Écriture dans la collection write via le service
  2. Émission d'un événement Socket.IO via EventsGateway
  3. Appel explicite au ViewBuilder pour projeter les changements dans le read model
@Post()
async create(@Body() body: any) {
  const item = await this.service.create(body);
  this.events.emit('eleve:created', item);
  this.viewBuilder.onEleveWrite();       // ← projection immédiate
  return item;
}

ViewBuilder

Le ViewBuilderService projette les données write dans les collections read.

Déclenchement

MécanismeQuandFiabilité
Appel expliciteAprès chaque POST/PATCH/DELETE dans un write controller✅ Toujours fiable (standalone ou replica set)
Change StreamsAutomatique sur chaque changement MongoDB⚠️ Requiert un replica set. Sur standalone, un warning est loggé et le mécanisme est désactivé.
Rebuild initialAu démarrage du serveur (onModuleInit)✅ Reconstruit tout à partir des collections write

Cascade de projection

Quand une entité change, le ViewBuilder reconstruit les collections read qui dénormalisent cette entité :

Méthode de projection

// Exemple : rebuildEleves()
const [eleves, classes] = await Promise.all([
  eleveModel.find().lean(),       // lecture write
  classeModel.find().lean(),
]);

const classesMap = new Map(classes.map(c => [c._id.toString(), c]));

const ops = eleves.map(e => ({
  updateOne: {
    filter: { source_id: e._id.toString() },
    update: { $set: {
      source_id: e._id.toString(),
      nom: e.nom, prenom: e.prenom, ...,
      classe_nom: classesMap.get(e.classe_id)?.nom || '',
    }},
    upsert: true,
  },
}));

// Supprimer les orphelins (documents read sans source write)
await readEleveModel.deleteMany({ source_id: { $nin: ids } });
// Upsert en bulk
await readEleveModel.bulkWrite(ops);

Pagination côté serveur

Toute la pagination et la recherche sont exécutées par MongoDB via find(filter).skip().limit(). Le frontend n'exécute plus de paginate(), filter() ou useMemo() pour les listes.

Exemple — recherche d'élèves

// Backend ReadService
async getElevesList(page = 1, limit = 12, search = '', classeId = '') {
  const filter: any = {};
  if (classeId) filter.classe_id = classeId;
  if (search) {
    filter.$or = [
      { nom: { $regex: search, $options: 'i' } },
      { prenom: { $regex: search, $options: 'i' } },
    ];
  }

  const [items, total] = await Promise.all([
    this.readEleveModel.find(filter).skip((page - 1) * limit).limit(limit),
    this.readEleveModel.countDocuments(filter),
  ]);

  return { eleves: items, total, page, limit, totalPages: Math.ceil(total / limit) };
}

// Frontend
const { data } = usePageFetch(
  useCallback(() => readApi.elevesList(page, 12, search, classeId), [page, search, classeId]),
);

Synchronisation temps réel

Après chaque mutation :

  1. Le write controller émet un événement Socket.IO (eleve:created, etc.)
  2. Le contexte React correspondant appelle notifyDataChange()
  3. Tous les hooks usePageFetch écoutent via onDataChange()
  4. Après un délai de 500ms (temps pour le ViewBuilder), ils re-fetchent leur endpoint read
  5. La page se met à jour avec les données fraîches

Le délai de 500ms entre le socket et le re-fetch garantit que le ViewBuilder a eu le temps de projeter les changements dans les collections read. Sur un système chargé, ce délai peut être augmenté.
