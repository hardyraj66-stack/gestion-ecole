🔐 Authentification & Comptes

Connexion JWT, gardes globales, rôles et gestion des comptes utilisateurs

Authentification par JWT (signature HS256, implémentation maison sans dépendance npm). L'API est protégée par défaut : toutes les routes exigent un jeton valide, sauf celles marquées @Public(). Le hachage des mots de passe utilise scrypt (intégré à Node).

Gardes globales

Enregistrées via APP_GUARD dans auth.module.ts, appliquées à toutes les routes dans l'ordre :

  • JwtAuthGuard — extrait Authorization: Bearer <token>, vérifie la signature et l'expiration, attache l'utilisateur à la requête ; laisse passer les routes @Public().
  • RolesGuard — vérifie que le rôle de l'utilisateur figure parmi ceux exigés par @Roles(...).

Jeton JWT

// Payload signé HS256
{ sub: userId, username, role, tv, jti, iat, exp }   // exp = iat + 12 h
//   tv  = tokenVersion (révocation globale)
//   jti = identifiant de session (appareil)

// Transmis par le client
Authorization: Bearer <token>

Sécurité — durcissement

  • Garde stateful — après vérification de signature, JwtAuthGuard recharge le compte en base et rejette (401) si actif=false, deleted=true ou si tv ne correspond plus à tokenVersion → désactivation / archivage / changement de mot de passe / « déconnexion de toutes les sessions » invalident immédiatement les jetons déjà émis.
  • Politique de mot de passe — minimum 8 caractères, au moins une lettre et un chiffre (validatePasswordStrength), appliquée à toute définition de mot de passe.
  • Soft-delete — « Supprimer » archive le compte (deleted=true, sessions révoquées), restaurable ; l'identifiant/email restent réservés.
  • Sessions par appareil — chaque login enregistre { jti, userAgent, ip } ; révocation individuelle ou en masse (tokenVersion++).
  • Anti brute-force — 5 échecs sur un identifiant → blocage 15 min ; en complément, limiteur par IP ciblé sur login/forgot-password/reset-password (auth-rate-limit.middleware.ts).
  • Anti-injection NoSQL — middleware global qui retire les clés $/. de body/query/params.
  • Mot de passe oublié — réponse neutre ; jeton à usage unique haché (SHA-256) expirant en 1 h ; lien envoyé par email.
  • Production — JWT_SECRET obligatoire et ≥ 32 caractères (refus de démarrer sinon), trust proxy derrière Caddy, MongoDB avec --auth (compte applicatif à privilèges limités), en-têtes de sécurité Caddy (HSTS, CSP…).
  • Audit — création / modification / suppression / restauration / réinitialisation des comptes journalisées (audit_logs).
  • Présence — la gateway Socket.IO diffuse presence:changed (qui est en ligne), affiché sur la page Utilisateurs.

Compte admin par défaut. Au premier démarrage, si aucun utilisateur n'existe, un compte admin / admin123 est créé automatiquement (surchargeable via ADMIN_USERNAME / ADMIN_PASSWORD). À sécuriser dès la première connexion. (En production, ce mot de passe par défaut a été réinitialisé en mot de passe fort avec changement forcé à la première connexion.)

Frontend — session & protection des routes

  • AuthContext — expose user, status (loading | authenticated | unauthenticated), login, logout, changePassword, hasRole(...). Au démarrage, restaure la session via GET /auth/me si un token est présent.
  • RequireAuth — redirige vers /login si non authentifié, vers /dashboard si le rôle ne fait pas partie de roles. La route /utilisateurs utilise roles={['admin']}.
  • httpClient — remplace window.fetch globalement : ajoute l'en-tête Authorization à chaque appel API et déclenche la déconnexion sur une réponse 401.
  • authStorage — token persistant en localStorage (clé auth_token), source de vérité unique partagée par le fetch, le socket et le contexte.
  • WebSocket authentifié — socketService ne se connecte que si un token est présent et le transmet dans le handshake (auth: { token }). La gateway EventsGateway.handleConnection vérifie le token et déconnecte toute connexion non authentifiée. refreshAuth() reconnecte (après login) ou coupe le socket (après logout).
  • Sidebar — affiche l'utilisateur connecté et son rôle, le bouton de déconnexion (avec confirmation) et le lien /utilisateurs visible uniquement pour les admin.
